# V6. 概率乘：质量是各环节成功率的乘积

## 1. 核心公理

一个多环节产出的最终质量，等于各环节成功率的**乘积**，不是平均值，也不是最差值之外可以忽略的小瑕疵。`0.99^10 = 0.90` 还行，`0.99^51 = 0.60` 已经不及格，`0.95^10 = 0.60` 直接崩。第一性推论：单环节追求"一次完美"是低效且脆弱的策略，因为概率乘会惩罚任何一个不完美的环节，而且惩罚随环节数指数增长。正确的策略是让**每个环节都可验证、可修复、可迭代**——把成功率从"听天由命"变成"被 eval 锚定"。这就是为什么"每个 action 必须对应一个 eval"不是工程洁癖，而是概率乘逼迫出的必然。

## 2. 深度推演

### 2.1 为什么是乘法不是加法

环节是串联的：下游默认上游正确，一个环节失败会让它之后的所有工作建立在错误基础上。人们习惯用"平均值"或"差不多就行"来估质量——"十个环节每个九成把握，总体应该还行"。但串联系统的可靠性是乘积：十个 0.9 相乘是 0.35，不是 0.9。平均值直觉只在环节彼此独立且可并行重试时成立，而真实流水线几乎都是串联依赖。乘法的残酷之处在于它没有"拖后腿的那个"以外的容错——一个 0.5 的环节能把九个 0.99 拖到 0.43。所以提升整体质量，杠杆点永远是**最弱的那个环节**，而不是已经很好的环节。

### 2.2 "一次完美"是反概率乘的策略

"让它一次做对"听起来高级，实际是赌单环节成功率接近 1。在 AI 协作里这几乎不可能——模型有抽卡率，长链任务里单次完美的概率相乘后趋近于零。概率乘给出的替代策略是：不追求单次完美，追求**每个环节可被快速发现错误、快速修复、可重复迭代**。一个成功率只有 0.7 的环节，如果能跑 N 轮自动修复（每轮独立），它的有效成功率会逼近 `1-(0.3)^N`——三轮就是 0.973。这比死磕"一次做对"现实得多。"多烧几轮"在概率乘下不是浪费，而是把弱环节的有效成功率拉上去的唯一手段。

### 2.3 action ↔ eval 对偶

概率乘的可操作推论：每个 action 对应一个可执行的 eval。action 没有 eval，它的成功率就是黑箱，n 个黑箱相乘必然崩。有 eval 的环节，成功率从"未知"变成"可观测、可收敛"。闭环形态：`action → eval → 不通过 → 自动修复（最多 N 轮）→ 仍不通过 → 升级人类`。N 的选择就是 acceptable threshold 的设定（见应用判定）：核心流程多烧几轮值得，探索性任务两轮即可。关键不是 eval 多严格，而是**每个 action 都有对应的 eval**——让概率乘的分母里没有黑箱环节。

### 2.4 与 X5 精度级联的区分

X5 讲的是**误差/精度**在耦合系统中级联放大，解药是精度预算、接口容差、降低耦合度。V6 讲的是**成功率**相乘决定最终质量，解药是 action-eval 对偶和迭代修复。两者都涉及"乘法"且都强调"每个接口处验证"，但关注点不同：X5 的对象是连续的精度量（0.5 像素偏差），V6 的对象是离散的成败（这一步过没过）。X5 说"上游不精确下游补不回来"，V6 说"成功率不可观测的环节会拖垮整体"。互补不重复：X5 告诉你接口处要设精度预算，V6 告诉你每个 action 要配 eval。

### 2.5 概率乘是可验证性地基的数学根基

V2 说"可验证性是信任的地基"。V6 给出**为什么**：不可验证的环节，其成功率既不可观测也不可控，n 个这样的环节相乘，整体成功率必然滑向不及格。可验证性把"成功率"从先验的赌博变成后验的度量——你能知道这一步过了没，于是能用迭代修复把有效成功率拉上去。换句话说，可验证性之所以是地基，正是因为没有它，概率乘就没有抓手：你既不知道哪个环节在拖后腿，也无法对它做有针对性的修复。V6 是 V2 的数学注脚。

### 2.6 反直觉推论：多烧 token ≠ 多花钱（缓存飞轮）

严格门禁会逼着 agent 反复访问同一段 context（修复同一个 action），这表面上"浪费 token"，实际触发正向飞轮：`严格门禁 → 同一段 context 被反复发送 → 高缓存命中 → 缓存命中的 token 近乎免费 → 迭代修复的实际成本坍塌`。所以"省 token"往往是反概率乘的——宽松门禁换来一次通过、低缓存命中、看似省了，实际质量差导致人工返工，总成本更高。本仓库实测（`tools/cache_hit_rate.py`）：长会话缓存命中率收敛到 95-98%，项目整体 96%。飞轮是否在转，是可观测的，不是信念。

## 3. 应用判定

**何时使用**：任何多环节、多步骤的产出——AI 协作流水线、多阶段数据处理、跨模块重构、CI/CD 链。只要"下游默认上游正确"成立，概率乘就在起作用。

**如何实践**：
1. 拆解流水线时，先识别环节数 n；n 越大，对单环节成功率要求越苛刻
2. 给每个 action 配 eval，让分母里没有黑箱环节（action ↔ eval 对偶）
3. 弱环节（成功率低、易错）配自动修复轮数 N，用 `1-(1-p)^N` 拉高有效成功率
4. 用缓存命中率观测飞轮健康度（`tools/cache_hit_rate.py`），命中率持续走低 = 门禁在松或 context 在碎
5. 别手动省 token——它往往在破坏飞轮

**acceptable threshold 参考**（因业务而异，关键是在期望预算/时间内出期望结果）：

| 场景 | 建议 threshold |
|------|---------------|
| 核心业务逻辑 / 安全 | 95%+ |
| 文档质量 | 90%+ |
| 探索性任务 | 70%+ |

**陷阱**：
- **黑箱环节**：没有 eval 的 action，成功率未知，是概率乘里的隐形炸弹
- **优化错环节**：给已经 0.99 的环节加投入，不如给 0.7 的环节加修复轮数
- **手动省 token**：破坏缓存飞轮，换来低质量和高返工总成本
- **追求一次完美**：在长链任务里概率趋零，不如追求每环可修复

## 4. 与其他公理的关系

- **V2 可验证性是信任的地基**：V6 是 V2 的数学根基——不可验证环节成功率不可控，相乘即崩
- **X5 精度在系统中级联**：同为乘法原理，X5 对象是精度量，V6 对象是成败，互补不重复
- **M1 闭环校准**：action-eval 对偶就是闭环校准的具体形态
- **A4 可靠性是管理问题**：概率乘解释了为什么把 AI 当工具（不管理每个环节的成功率）必然不可靠
- **X3 效率由瓶颈决定**：概率乘的杠杆点是最弱环节，与瓶颈决定效率同构

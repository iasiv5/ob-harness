# ob 优先：OpenBMC 环境动作的统一前门

## 元数据

- **类型**: BestPractice
- **适用场景**: agent 准备做任何 OpenBMC 环境生命周期动作（初始化、编译、状态查询、QEMU 起停等）之前
- **创建日期**: 2026-06-20

---

## 目标

agent 做 OpenBMC 环境动作时，以 `ob` 为唯一前门：先查 `ob` 是否提供该能力，提供就调用 `ob <cmd>`，仅在真实失败且 `ob` 确无此能力时才手动兜底。这样能力随 `ob` 集中演进，避免 agent 各自手搓导致行为发散、难维护。

## 适用边界

- **适用**：OpenBMC 环境生命周期动作——`ob --help` 列出的子命令所覆盖的意图（当前 `init` / `build` / `status` / `start-qemu` / `stop-qemu`，未来随 `ob` 增长，如规划中的 `ob dev`）。
- **不适用**：`ob` 显然不涉足的动作（手改 recipe、解析 D-Bus 日志、写 skill、分析 BitBake 日志等）。这些是 agent 自由发挥区，不必先问 `ob`。
- **判定入口**：`ob --help`。凡 help 列出的子命令覆盖的意图，一律走 ob 优先；help 没有的，才是自由区。

## 能力发现：`ob --help` 是唯一权威清单

- `ob --help`（`usage()`）是 `ob` 能力的唯一权威清单，随 `ob` 增长更新；`tests/protocol/usage_dispatch_sync.sh` 断言它与 dispatch 子命令集合一致，防止漂移。
- 不要凭记忆假设 `ob` 有或没有某能力——`ob --help` 现查现用。
- planned 但未发布的命令（如规划中的 `ob dev`）**不进** `--help`；发现规则只回答"现在能调什么"，不回答路线图。

## exit-code 契约

`ob` 所有 `cmd_*` 子命令统一的退出码语义。agent 据此判断"是否回退"：

| 码 | 语义 | agent 处置 |
|----|------|-----------|
| **0** | 成功，或良性无操作（如无实例可停、dry-run 完成） | 完成，继续 |
| **1** | 真实失败：`ob` 坏了，或用法错误（未知选项、stdin 不可读等） | 唯一触发"诊断 → 改根因 → 必要时手动"的码 |
| **2** | 用户主动取消（选 Q / 输 0 / 拒绝确认 banner）——不是失败 | 停下回报，不重试、不转手动 |
| **3** | 前置缺失（缺 machine / 无 TTY / 未 init / 未 build），修复方式是用 `ob` 补前置 | 按 `ob` 提示补前置再重试，**不要据此转手动** |

## 按码回退协议

1. **exit 0** → 完成。
2. **exit 2** → 用户取消，停下回报，不重试、不转手动。（非交互场景几乎只在确认被 wrapper 拒时出现；用显式参数 + `--force` 可绕过 prompt。）
3. **exit 3** → 读 `ob` 的 error 提示，用 `ob` 补前置再重跑（典型链：`ob init <machine>` → `ob build` → `ob start-qemu`），或补全 CLI 参数。**禁止**因 exit 3 转手动。
4. **exit 1 / 其他** → 真实失败才触发回退：先读 `ob` 自己的报错 + 相关 `ob` 源码定位根因 → 根因在 scope 内就改掉重跑 `ob` → **只有当 `ob` 确实没这能力、或 `ob` 自身坏在 agent scope 之外时，才手动兜底**。

## 手动兜底：严格度与确认

- 复用 `rules/01_SOUL.md` 的"浮出来等用户"框架：手动兜底若触及不可逆 / 越界 / 需要用户领域判断三类点 → **先确认**；否则直接做。
- 无论是否需要确认，手动绕过 `ob` 后**必须**：(1) 记录绕过了哪个 `ob` 能力 + 原因；(2) 把"`ob` 缺这能力"列为 `ob` 待补项。这条反馈环让缺失能力回流进 `ob`，使 `ob` 越长越全（对齐公理 A12 AI 原生开发范式）。

## 验收标准

一个无上下文 agent 据此即可自检是否守约：

1. 动手 OpenBMC 环境动作前，是否先查过 `ob --help`？
2. 命中 `ob` 能力时，是否先 `ob <cmd>` 而非手搓？
3. 遇 exit 2 / exit 3 时，是否**没有**错误地转手动绕过 `ob`？
4. 真实失败（exit 1）确需手动兜底时，是否记录了绕过 + `ob` 待补项？

## 已知缺口

- **`ob build` 当前纯交互**：非 TTY 直接 `exit 3`（"ob build requires interactive mode"），没有 `ob build <machine>` 非交互路径。agent 在非交互终端下遇到时，按 exit 3 协议处理——**不要转手动跑裸 `bitbake`**，记为 `ob` 待补项（未来给 `build` 加非交互路径）。

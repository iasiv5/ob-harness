# Spec Self-Review

写完设计文档后，按下面的顺序做一轮 inline 自检，只修会影响后续计划的问题。

## 1. 占位符扫描

检查是否存在以下问题：
- `TODO`、`TBD`、`later`、`待补充`
- 空章节
- 含糊的“后续再看”表述

如果存在，直接补全或删掉。

## 2. 内部一致性

检查：
- 推荐方案是否和前面的方案比较一致
- 组件边界是否和后文数据流匹配
- 错误处理是否和范围定义冲突

## 3. 范围检查

检查这份设计是否足够聚焦，能交给单份实施计划：
- 如果覆盖多个独立子系统，先拆分
- 如果出现明显超范围功能，删掉或移到非范围

## 4. 歧义检查

检查是否有要求会被实现者理解成两种不同做法：
- 命名是否清楚
- 输入输出是否清楚
- 成功标准是否清楚

如果存在歧义，明确写死一种解释。

## 5. YAGNI 检查

删除这些典型过度设计：
- 用户没要求的扩展能力
- 只是“以后可能需要”的抽象层
- 和当前目标无关的泛化接口

## 通过标准

满足以下条件即可进入用户审批：
- 没有占位符
- 没有关键矛盾
- 范围适合单份实施计划
- 不存在会导致计划写偏的关键歧义

# Questioning Rules

澄清阶段必须遵守这些规则。

## 核心规则

- 每条消息只保留一个待回答的问题
- 优先使用单选题，其次多选题，最后才是开放题
- 不要在解释、总结或推荐后面偷偷追加第二个问题
- 如果一个话题需要继续深挖，也要拆成下一轮再问

## 优先澄清的维度

按这个顺序逐步问：
1. 目标：为什么要做这件事
2. 约束：性能、兼容性、依赖、上线边界、交互方式
3. 成功标准：做成什么样才算完成

## 适合的提问方式

优先这样问：
- "这次更偏向哪一种？A / B / C"
- "这次的首要目标是 1 / 2 / 3 哪个？"
- "下面两个限制，哪个必须优先满足？"

只在必要时才这样问：
- "你理想中的体验是什么？"
- "有没有额外约束还没说？"

## 什么时候停止追问

满足以下条件就不要继续追问，进入方案比较：
- 已经知道目标
- 已经知道关键约束
- 已经知道成功标准

如果请求本身跨多个独立子系统，也不要继续抠细节，先做拆分。

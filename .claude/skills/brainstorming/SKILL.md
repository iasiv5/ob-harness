# Brainstorming Skill

这个 skill 用于把模糊想法收敛成经用户批准的设计文档，再把下一步明确交给 `writing-plans`。

这是一个 manual-first 的规划 skill，不是实现 skill。
在这个 skill 中，设计获批前不要写代码、不要生成补丁、不要进入实现。

## 来源致谢

这个中文化适配版本致敬 obra/superpowers 的 brainstorming skill：
https://github.com/obra/superpowers/tree/main/skills/brainstorming/

当前实现不是逐字搬运，而是面向 GitHub Copilot 的中文化、manual-first、artifact-backed 适配版。

## 何时使用

在以下场景优先使用这个 skill：
- 用户想新增功能、调整行为、设计组件、设计接口或改变系统流程
- 用户想先比较几种实现方案，再决定怎么做
- 用户只给了目标和约束，但还没有清晰设计
- 任务涉及多个文件、多个边界，或存在明显取舍
- 任务看起来很小，但仍需要先确认范围、成功标准或边界

## 不要路由到 brainstorming

以下请求默认不要使用这个 skill：
- 已经有批准过的设计文档，只是要写实施计划
- 直接实现、修 bug、改代码、跑测试或提交补丁
- 代码审查、缺陷分析、交接总结、状态同步
- 单纯解释某段代码、回答事实问题，或查找定义

## 硬约束

- 设计获批前，不要在本 skill 中写代码、生成补丁或切到实现流程
- 一次只问一个问题；如果需要继续追问，也要等用户回答后再问下一题
- 优先使用单选或多选问题；只有在必须时才使用开放题
- 先看当前项目上下文，再提设计建议；跟随现有模式，不做无关重构
- 如果需求跨多个独立子系统，先拆分，再只处理第一个子项目
- 设计文档默认要落盘；路径优先级是：用户指定路径 > 仓库已有约定 > `docs/specs/`
- 写完设计文档后先做 inline 自检，再让用户审批
- 这个 skill 的终态是进入 `writing-plans`，不是直接编码

## 🔴 CHECKPOINTS

- 🔴 CHECKPOINT A · 方案方向确认：只有在用户明确接受推荐方案、范围和关键边界后，才能进入 PHASE 5 写设计文档
- 🔴 CHECKPOINT B · 文档审批：只有在用户明确批准设计文档后，才能进入 PHASE 8 和 `writing-plans`
- 🛑 STOP · 错误路由：如果请求本质上是实现、修 bug、代码审查，或已有设计后的实施计划，立即停止本 skill，改走对应流程

## 失败模式与回退

| 触发条件 | 一线处理 | 仍失败时 |
|---|---|---|
| 请求本质不是设计 | 明确停止 brainstorming，并路由到实现、review 或 `writing-plans` | 如果用户仍坚持继续，先要求用户指出仍未定的设计问题；如果说不出来，就不要伪造设计流程 |
| 相关上下文找不到，或仓库里没有现成实现可参考 | 明确缺哪些事实，列出 1-3 条工作假设，再只问一个选择题补齐关键约束 | 按最保守假设输出边界级设计，并把风险写进未决事项 |
| 需求跨多个独立子系统 | 先拆成子项目并说明依赖顺序，只继续第一个子项目 | 如果用户要求一次覆盖全部，只写父级设计和拆分清单，不把它伪装成单份可实施设计 |
| 用户回答过宽、同时改变多个前提，或没有真正回答问题 | 把回答收束成单选或多选确认，继续保持一次只问一个问题 | 如果仍然含糊，显式写下当前采用的工作假设，再进入方案比较 |
| 用户要求直接写文档或实施计划，但方案尚未确认 | 先给推荐方案和 trade-offs，然后触发 CHECKPOINT A | 如果用户只要草稿，文档标题显式标注“草稿 / 待审批”，且不要进入 `writing-plans` |

## 默认响应模板

优先用下面的固定形态，减少首轮回复漂移。

### 模板 1：首轮澄清

- 第一段只做一件事：说明先把需求收敛成设计，不进入实现
- 如果请求跨多个独立子系统，第二段先列出子项目，并明确“这轮只继续第一个子项目”
- 最后一段只保留一个问题，默认写成 A / B / C 选择题
- 如果目标、关键约束、成功标准还没钉住，不要在首轮消息里提前展开完整方案比较

默认形态：

> 先把这次讨论收敛成设计文档，不进入实现。
> 这次需求可以拆成 X / Y / Z 三块；这轮我只继续 X。
> 先确认一个点：这次的首要目标更偏 A / B / C？

### 模板 2：方案比较

- 用固定标题 `方案 A`、`方案 B`、`方案 C`
- 每个方案只写 4 项：核心思路、优点、缺点、适用条件
- 最后单独写 `推荐方案`，说明推荐原因和主要 trade-offs
- 收尾只问一个确认问题：接受推荐方案，还是改选 A / B / C

默认形态：

> 方案 A：……
> 方案 B：……
> 方案 C：……
> 推荐方案：……
> 先确认一个点：你接受推荐方案，还是要改选 A / B / C？

### 模板 3：错误路由

- 已经有批准过的设计，或用户明确只要实施计划时，直接停掉 brainstorming
- 路由信息要写清楚“为什么不该继续”以及“下一步要什么输入”

默认形态：

> 这一步不该继续用 brainstorming，应该切到 `writing-plans`。
> 你现在要的是把已定需求拆成实施计划，不需要再做方案比较。

### 模板 4：审批请求

- 文档写完并自检后，直接请求审批，不追加第二个问题
- 只说明文档路径、审批动作和审批后下一步

默认形态：

> 设计文档已写好并保存到 `<path>`。请先审阅这份设计；如果需要改动，我会先改设计，再进入实施计划。

## PHASE 0: 验证请求

开始前先确认：
- 这次请求确实需要设计，而不是单纯实现或审查
- 用户真正想解决的问题、约束和成功标准是什么
- 当前请求是否过大，是否应该先拆成多个子项目

如果需求明显包含多个独立子系统：
- 先指出拆分必要性
- 帮用户明确子项目边界、依赖关系和优先顺序
- 只对第一个子项目继续完整设计流程

## PHASE 1: 探索上下文

先读取最小必要的项目上下文：
- 相关文件、文档、已有接口、命名模式
- 与当前需求直接相关的调用点或测试
- 最近改动中会影响本设计的事实

探索目标不是把项目全部看完，而是回答三个问题：
- 当前系统怎么组织这块能力
- 哪些边界已经存在，应该沿用
- 哪些现实约束会影响设计选择

## PHASE 2: 一次一个问题地澄清

先阅读 [references/questioning-rules.md](references/questioning-rules.md) 里的规则，再发澄清问题。

澄清阶段的目标：
- 明确目的：为什么要做这件事
- 明确约束：性能、兼容性、时间、依赖、交互方式、发布边界
- 明确成功标准：什么算完成，什么算超范围

规则：
- 每条消息只保留一个待回答的问题
- 如果可以做成选择题，就不要发散成开放题
- 不要在同一条消息里偷偷附带第二个问题
- 首轮澄清默认套用“模板 1：首轮澄清”

## PHASE 3: 提出 2-3 个方案

在你理解需求后，给出 2-3 个方案：
- 每个方案说明核心思路、主要 trade-offs、适用条件
- 优先给推荐方案，并说清推荐原因
- 用 YAGNI 约束设计，不要添加用户没要的复杂度
- 如果现有代码边界已经足够好，优先复用；不要为了“更漂亮”而重做结构
- 输出默认套用“模板 2：方案比较”

## PHASE 4: 分段呈现设计并确认

设计要分段呈现，并在每个重要段落后确认：
- 架构与边界
- 关键组件或模块职责
- 数据流或控制流
- 错误处理与回退策略
- 测试策略

每段内容要和复杂度匹配：
- 简单问题可以是几句话
- 复杂问题可以是更完整的段落

如果用户对某一段提出异议：
- 回到相关问题重新澄清
- 修正设计后再继续下一段
- 每一段收尾只保留一个确认动作，不要顺手开启下一段

🔴 CHECKPOINT A：用户没有明确接受推荐方案、范围和关键边界前，不要进入 PHASE 5。

## PHASE 5: 写设计文档

在设计得到口头确认后，把设计写成文档。

路径选择顺序：
1. 用户显式指定路径
2. 仓库已有设计文档约定
3. 默认写到 `docs/specs/<YYYY-MM-DD>-<topic>-design.md`

如需固定骨架，先读取 [references/spec-template.md](references/spec-template.md)。

设计文档至少覆盖：
- 背景与目标
- 范围与非范围
- 方案比较与推荐方案
- 关键边界、组件和职责
- 数据流或控制流
- 错误处理
- 测试策略
- 未决事项

## PHASE 6: Inline 自检

写完文档后，读取 [references/spec-self-review.md](references/spec-self-review.md) 并完成单轮 inline 自检。

修复真正会影响后续计划的问题：
- TODO、TBD、占位符
- 内部矛盾
- 范围过大
- 含糊到会导致写错计划的要求
- 明显过度设计

不要为了文风润色反复循环。修完就继续。

## PHASE 7: 用户审批

自检通过后，明确让用户审阅设计文档。

建议使用类似表述：

> 设计文档已写好并保存到 `<path>`。请先审阅这份设计；如果需要改动，我会先改设计，再进入实施计划。

如果用户要求修改：
- 修改文档
- 重新跑同一轮 inline 自检
- 再次请求审批
- 审批请求默认套用“模板 4：审批请求”

只有在用户明确批准后，才能进入下一阶段。

🔴 CHECKPOINT B：没有明确批准，就停在设计阶段；不要切到 `writing-plans`。

## PHASE 8: 过渡到 writing-plans

用户批准设计后：
- 明确说明下一步是 `writing-plans`
- 把已批准的设计文档路径作为稳定输入
- 不要在本 skill 中直接转入实现
- 如果用户只批准了方向，没有批准文档，回到 PHASE 5-7，不要跳过审批
- 如果请求本质上是实施计划，默认套用“模板 3：错误路由”

## 反例黑名单

不要这样做：
- 一条消息里塞多个澄清问题，或在推荐后偷偷追加第二个问题
- 目标、约束、成功标准还没钉住，就直接给完整设计或长篇方案比较
- 把跨多个子系统的请求包装成一份巨大设计文档，假装它已经可实施
- 设计未批准就写代码、补丁、测试步骤或实现细节
- 为了“更干净”而重做现有边界、命名或仓库结构
- 没有最小必要上下文时假装已理解系统，而不是显式声明工作假设和未决事项
- 路由到 `writing-plans` 时，重新发起一轮需求澄清或方案比较

## 关键原则

- One question at a time：一次只问一个问题
- Multiple choice first：优先选择题
- YAGNI：不把未请求功能塞进设计
- Alternatives before commitment：先比较方案，再定设计
- Incremental validation：分段确认，不把整份设计一次性砸给用户
- Follow the repo：优先跟随现有仓库模式和边界

## 立即执行

先验证请求与范围，再读取最小必要上下文，然后按上述流程推进到已批准的设计文档。
